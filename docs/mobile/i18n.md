## ðŸ“¡ API error messages (shared)

Backend error responses include a `code` (e.g. `ERROR_INVALID_CREDENTIALS`). Use **`@fitnexa/shared/i18n`** so these show in the user's language.

**Option A â€“ Merge into i18next (recommended):** After initialising i18next, add the shared API error bundle so `t('errors.ERROR_XXX')` works:

```ts
import { apiErrorResources, API_ERROR_LOCALES } from '@fitnexa/shared/i18n';

API_ERROR_LOCALES.forEach((locale) => {
  i18n.addResourceBundle(locale, 'translation', apiErrorResources[locale], true, true);
});
```

Then when handling API errors: `const message = data?.code ? t(\`errors.${data.code}\`) : data?.error;`

**Option B â€“ Standalone:** `import { getApiErrorMessage } from '@fitnexa/shared/i18n';` then `getApiErrorMessage(data?.code, currentLocale, data?.error)`.

See `@fitnexa/shared` package, `src/i18n/README.md`, for full API.

## ðŸš€ Usage in Mobile Components

We use custom hooks to wrap translation logic:
```typescript

  const { t } = useTranslation();
  return {
    title: t('home.welcome_back'),
    // ...
  };
};
```
